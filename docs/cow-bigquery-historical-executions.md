# Historical finalized CoW execution backfill

## Scope

`orcap.cow_history` reads the public, partitioned
`bigquery-public-data.crypto_ethereum.logs` table for exact
`GPv2Settlement.Trade` and `GPv2Settlement.Settlement` event logs. The CoW
[contract specification](https://docs.cow.fi/cow-protocol/reference/contracts/core/settlement)
identifies `Trade` and `Settlement` as indexed events and defines a settlement
as a batch of trades. The result is a finalized on-chain event panel, not an
order-submission feed.

The collector deliberately has no GitHub workflow or Hugging Face path. It
only writes `data/external/cow_bigquery_history/` locally. Running it requires
Application Default Credentials with a BigQuery billing/quota project. Public
dataset storage is free, but query processing is billed to that project.

## Safety contract

1. Callers provide an explicit, UTC, end-exclusive date range.
2. A request may cover at most 14 days and is dry-run first.
3. The dry run must be at or below a 50 GiB `maximum_bytes_billed` cap before
   an actual job is submitted.
4. The retained manifest includes the public table, query SHA-256, requested
   range, projected and actual bytes processed, run time, and raw-row count.
5. Execution requires `--execute`; omitting it prints only the cost estimate.
   Neither form uploads or enables scheduling.

For example, after an explicit authorization to bill the configured BigQuery
project:

```bash
PYTHONPATH=. .venv/bin/python -m orcap.cow_history \
  --start 2026-07-01 --end 2026-07-08

PYTHONPATH=. .venv/bin/python -m orcap.cow_history \
  --start 2026-07-01 --end 2026-07-08 --execute
```

The public table is day-partitioned but not clustered by contract address. A
longer date range can therefore be expensive even with a selective event
filter; the short hard limit is intentional. A 2025-01-01-to-2026-07-10 dry
run measured 943.5 GiB on 2026-07-10, so that range is not eligible for one
job.

## Normalized fields and boundaries

The raw log parquet is retained alongside two normalized tables:

- **executions:** one record per finalized `Trade`, with raw token amounts;
  only registered mainnet USDC/WETH pairs receive a USDC/WETH price unit. A
  `Settlement` solver is joined only when it has the same transaction hash.
- **events:** every retained `Trade` and `Settlement` log, keyed by immutable
  transaction hash and log index.

`Trade` events are individual fills inside a batch. They are not submissions,
market-wide demand, routed requests, price improvement, all-in cost, user
surplus, or solver market share. A settlement transaction gives neither an
auction's submitted orders nor a realized allocation rule.

## H65 arrival comparator

`orcap.analysis.h65_cow_settlement_arrivals` evaluates **one `Settlement`
event per transaction** in non-overlapping 15-minute UTC bins. It uses a
chronological holdout to compare a constant-rate Poisson baseline, a smoothed
UTC-hour Poisson baseline, and a Poisson-INGARCH(1,1), which is only a
discrete-time exponential-Hawkes analogue. It expressly does not use `Trade`
count as the arrival unit because batch size is mechanically generated by a
settlement contract and would overstate apparent self-excitation.

```bash
PYTHONPATH=. .venv/bin/python -m orcap.analysis.h65_cow_settlement_arrivals \
  --events data/external/cow_bigquery_history/cow_bq_2026-07-01_2026-07-08_events.parquet \
  --out /tmp/h65
```

H65 is only descriptive after at least seven complete UTC days and 100
settlements. Better held-out likelihood for INGARCH than **both** Poisson
baselines would support only the narrow statement that this binned finalized-
settlement series is better predicted by the specified count-autoregressive
model than those fixed-rate alternatives. It is not evidence of a continuous-
time Hawkes process, strategic self-excitation, adverse selection, demand
bursts, or an analogue of model-router allocation.

## First local panel — 2026-07-10

The first authorized, local-only run covered 2026-07-03 through 2026-07-09
(end exclusive: 2026-07-10). Its dry run projected 20.745 GiB and the
completed job processed 20.323 GiB, below the 50 GiB hard cap. It retained
44,803 source logs: 22,660 `Trade` records and 22,143 `Settlement` records.
No result was uploaded to Hugging Face.

H65 formed 672 complete 15-minute bins over the seven full UTC days. The
settlement-count Fano factor was 4.148. In the chronological 470-bin/202-bin
training/test split, the smoothed UTC-hour Poisson improved over a constant
Poisson by 4.762 held-out log-likelihood points. The specified INGARCH
improved over the UTC-hour baseline by 70.471 points (persistence 0.897).

This clears H65's deliberately modest descriptive coverage gate, but it does
not identify a continuous-time Hawkes process or a strategic mechanism. The
within-panel 5-minute, 15-minute, 30-minute, and 1-hour grid across 60%, 70%,
and 80% training splits favored INGARCH in 11 of 12 specifications. The
exception was the 1-hour/80% training split (INGARCH-minus-diurnal held-out
log likelihood −15.900); the range was −15.900 to 108.335. Thus it is not
valid to call the ranking bin- and split-robust.

The remaining checks are a longer preregistered window, richer
seasonality/exogenous controls, and a direct router-arrival panel with
comparable event semantics. Until then, the defensible statement is only that
the preregistered 15-minute/70% split of this short finalized-settlement series
is materially better predicted by the declared binned count-autoregressive
model than the two fixed-rate Poisson baselines.
